# 01 · 시스템 구성

> 일짜곰은 프론트엔드(웹), AI 오케스트레이션 백엔드, 통합 데이터 플랫폼(Supabase), 외부 생성 AI(Gemini·Imagen)로 이루어진다. 백엔드가 모든 AI 흐름을 조율하고, 프론트는 그 결과 스트림을 소비한다.

---

## 1. 구성 요소

일짜곰은 네 개의 큰 구성 요소로 나뉜다.

| 구성 요소 | 기술 | 역할 |
| --- | --- | --- |
| 프론트엔드 | Next.js 16 (웹) | 학생·교사·관리자 화면. 백엔드 API 소비, 본문 스트림 렌더. 데이터 격리는 백엔드에 위임하고 화면·라우팅만 담당. |
| 백엔드 | Python FastAPI | AI 오케스트레이션 허브. 4계층 AI 파이프라인 + 이미지 생성 조율, 인증 검증, 접근제어, 데이터 영속화. |
| 데이터 플랫폼 | Supabase (PostgreSQL + pgvector / Storage / Auth) | 도메인 데이터·관측 데이터 저장, 벡터 검색(RAG), 삽화 파일 저장, Google OAuth 인증. RLS로 행 단위 격리. |
| 생성 AI | Google Gemini 2.5 (Pro·Flash·Flash-Lite), Imagen 4 | 설계·집필·편집·대화 본문 생성과 삽화 생성. |

핵심 설계 원칙은 다음과 같다.

1. **AI 오케스트레이션 허브**: 백엔드가 AI 파이프라인을 조율하고 프론트는 스트림만 소비한다.
2. **DB 단일화**: 인증·도메인 데이터·벡터·스토리지를 Supabase로 통합하고, RLS로 학급·학교 단위로 격리한다.
3. **비용 안정성**: Bible와 RAG로 호출당 토큰을 고정하고, 모델 등급을 호출 빈도에 맞춘다(Pro 1회 / Flash 반복 / Flash-Lite 대화).
4. **스트리밍 우선**: 집필 본문은 SSE로 글자 단위 전송해 생성 대기 중에도 체감 지연을 줄인다.
5. **안전 기본 통로**: 모든 입력과 출력이 안전 게이트를 통과한다(미성년 전제).

## 2. 요청 흐름

브라우저의 요청은 항상 백엔드를 거쳐 Supabase와 Gemini/Imagen에 도달한다. 프론트가 외부 AI나 DB를 직접 호출하지 않는다.

```
[브라우저 / Next.js 16]
        │  Authorization: Bearer <Supabase access token>
        ▼
[FastAPI 백엔드]
   ├─ deps.py        인증(JWT 검증) · 역할 가드 · 보호자 동의 게이팅
   ├─ api/*          라우터(books · chapters · planning · teacher · admin · ai · safety · events)
   ├─ services/*     비즈니스 로직 + 접근제어(RLS 등가)
   ├─ ai/*           4계층 파이프라인 · Skill/ReAct · RAG · 안전 게이트
   └─ store/*        Store 추상 → SupabaseStore | InMemoryStore
        │                          │
        ▼                          ▼
[Gemini 2.5 / Imagen 4]    [Supabase Postgres + pgvector / Storage / Auth]
```

대표 흐름(학생 챕터 읽기)은 다음과 같다.

1. 프론트가 `GET /books/{id}/chapters/{idx}/stream`을 SSE로 구독한다.
2. 백엔드가 토큰을 검증하고(`deps.py`), 서비스 계층이 책 접근 권한을 확인한다.
3. 집필 AI가 RAG로 Bible의 관련 항목을 인출해 컨텍스트를 고정한 뒤 본문을 생성한다.
4. 생성된 본문 조각이 `token` 이벤트로 흘러 나가고, 완료 시 `done` 이벤트가 전송된다.
5. 본문·청크·임베딩이 Supabase에 저장되어 다음 구독부터는 저장본을 그대로 흘린다.

---

## 3. SSE 스트리밍

집필 본문은 표준 SSE(`text/event-stream`)로 전송된다. FastAPI의 `StreamingResponse`(async generator)가 토큰 조각을 흘리고, 프론트는 `token`을 순서대로 이어 붙인다.

| 특성 | 설명 |
| --- | --- |
| 이벤트 종류 | `meta`(시작 메타) · `illustration`(삽화) · `prompt`(유도 질문) · `ask`(분기 질문) · `token`(본문 조각) · `done`(종료) · `error`(오류) |
| 저장본 제공 | 최초 구독 시 생성·저장하고, 이후 재구독은 저장된 본문을 그대로 흘린다. 편집·수정 반영분이 포함된다. |
| 재연결 | 쿼리 `?from=<charOffset>`로 이어받기를 지원한다. `error(retryable)` 또는 비정상 종료 시 재구독한다. |
| 하트비트 | 서버가 15초마다 `: ping` 코멘트를 보내 연결을 유지하고, 버퍼링 방지 헤더(`X-Accel-Buffering: no`, `Cache-Control: no-cache`)를 둔다. |

생성 지연 자체는 다음 장 백그라운드 선생성으로 흡수해 프록시 idle cut과 끊김을 줄인다.

## 4. 무상태 · 배포 준비

백엔드는 무상태(stateless)로 설계되어 여러 워커로 수평 확장할 수 있다. 프로세스 메모리에 의존하던 상태는 모두 DB로 옮겼다.

- **무상태 rate limit**: 호출 한도 카운터를 프로세스 메모리가 아닌 `rate_hits` 테이블에 저장해, 멀티 워커에서도 한도가 정합한다.
- **컨테이너화**: `Dockerfile`로 이미지화하며, 상세는 `backend/DEPLOY.md`를 따른다.
- **헬스체크**: 두 종류를 제공한다.

| 엔드포인트 | 용도 | 동작 |
| --- | --- | --- |
| `GET /health` | 라이브니스 | 외부 의존 없이 빠르게 응답. 저장소·AI 모드를 표기. 운영인데 인메모리/mock로 떨어지면 `status: degraded`. |
| `GET /health/ready` | 레디니스 | Supabase 연결을 가벼운 쿼리로 1회 확인. 실패 시 503. |

- **상시 가동**: 백엔드는 항상 떠 있으며, 백그라운드 알림 루프가 `app_settings.notify_interval_sec`(기본 180초) 주기로 안전 신호·오류 세션을 점검해 관리자 알림을 생성한다.

## 5. 환경 모드

실행 모드는 `APP_ENV`(`dev` / `test` / `prod`)로 구분되며, 모드에 따라 저장소와 인증 정책이 달라진다.

| 모드 | 저장소 | AI | 인증 | 비고 |
| --- | --- | --- | --- | --- |
| `dev` | Supabase 또는 인메모리 | 키 있으면 실 AI, 없으면 mock | Supabase JWT 또는 dev 토큰 | 개발 편의로 인메모리 폴백 허용(경고). |
| `test` | 인메모리(명시 허용) | mock(결정적) | dev 토큰 | 계약 검증용. 결정성 위해 백그라운드 루프 비활성. |
| `prod` | Supabase(필수) | 실 AI | Supabase JWT만 | 자격 없으면 기동 거부(fail-closed). 인메모리/Noop 폴백 금지. |

폴백 동작의 기준은 자격(credential) 유무다.

- **저장소**: `SUPABASE_URL`·`SUPABASE_SERVICE_ROLE_KEY`가 있으면 `SupabaseStore`, 없으면(비-prod) `InMemoryStore`.
- **AI**: `GOOGLE_API_KEY`가 있으면 실제 Gemini/Imagen 호출, 없으면 결정적 mock 응답.

이 폴백 구조 덕분에 외부 키 없이도 API·SSE 계약이 그대로 동작한다. `GET /health`로 현재 모드(`storage`, `ai`)를 확인할 수 있다.

안전장치로, 운영 신호가 켜진 상태에서 개발 인증이 활성화되어 있으면 앱 기동을 거부한다. 즉 `DEV_AUTH=true`인데 `SUPABASE_JWT_SECRET` 또는 `SUPABASE_URL`이 설정되어 있으면 설정 실수로 보고 기동을 막는다(fail-closed).
