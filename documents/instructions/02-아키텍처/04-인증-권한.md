# 인증 / 권한

> Google 로그인(Supabase Auth) + 백엔드 JWT 검증 + Postgres RLS의 3중 구조. 미성년 서비스라 보호자 동의·안전강도 계층이 더해진다.

## 1. 로그인 흐름 (Google OAuth)
```
브라우저 /login → Supabase signInWithOAuth(google, prompt=select_account)
  → Google 동의 → /auth/callback?code=... → 코드↔세션 교환(@supabase/ssr)
  → 역할별 홈으로 리다이렉트
```
- `prompt=select_account`로 항상 계정 선택을 띄운다(공유 기기 학교 환경 대비, 자동 재로그인 방지).
- 프론트는 `proxy.ts`(Next 16)에서 매 요청 세션 쿠키를 갱신하고 비로그인 사용자를 `/login`으로 보낸다(낙관적 가드). 권위 있는 역할 판별은 역할별 라우트 그룹 layout에서 `GET /me`로 한다.

## 2. 백엔드 토큰 검증 (ES256 / JWKS)
Supabase가 발급하는 사용자 access_token은 **`alg=ES256`(비대칭, kid 포함)**이다. 백엔드는 JWKS로 비대칭 검증한다.
- JWKS: `{SUPABASE_URL}/auth/v1/.well-known/jwks.json` (PyJWKClient, URL당 캐시)
- 검증: 서명 + `iss={SUPABASE_URL}/auth/v1` + `aud=authenticated` + 만료
- 레거시 HS256(`SUPABASE_JWT_SECRET`)도 과도기 호환. **검증 실패는 전부 401 fail-closed**(만료·위조·aud/iss·형식·JWKS 네트워크 오류 모두 401, 절대 500 아님).
- 개발 모드(`DEV_AUTH=true` + JWT 시크릿 없음): `Authorization: Bearer dev:<email>:<role>` 허용. 시크릿이 있으면 dev 토큰은 무시되고 기동도 거부(fail-closed).

## 3. 역할과 RLS
- 역할: `student` · `teacher` · `admin`(`profiles.role`). 관리자는 `ADMIN_EMAILS` 화이트리스트 + role 둘 다 통과해야 한다.
- 모든 데이터 접근은 Postgres **RLS**로 DB 차원에서 격리된다. 헬퍼 `is_admin()`, `can_access_book(book_id)`.
  - 학생: 자기 책/챕터/학습/편지/협업만.
  - 교사: 자기 학급(`classrooms.teacher_id`)에 속한 책·학생만.
  - 관리자: 전체.
- 최초 역할 선택(온보딩) 후에는 재온보딩으로 역할을 바꿀 수 없다(권한 상승 방지).

## 4. 보호자 동의
만 14세 미만 전제로 보호자 동의(`profiles.guardian_consent`)를 받는다. 미동의 학생은 책 생성·기획·편지·수정 등 생성 동작이 `consent_required`(403)로 차단된다.

## 5. 안전강도 계층
AI 콘텐츠 필터 세기(`safety_level`: `relaxed|standard|strict`)는 다음 순서로 해석된다.
```
발제(prompt) > 학급(class_settings) > 전역(app_settings, 기본 standard) > 코드 기본
```
교사가 학급별로 조정할 수 있고(`class_settings`), 관리자가 전역 기본을 선택형으로 설정한다. 집필·삽화 호출 시 해석된 값이 Gemini/Imagen 안전 설정에 반영된다.

## 6. rate limit / 감사
- 사용자별 호출 한도(60초 윈도)를 무상태로 적용(초과 시 `429 rate_limited`). 읽기 스트림은 무제한.
- 관리자 행위는 `audit_log`에 기록된다.

## 7. 학급 가입 코드
교사가 만든 학급의 `code`(CSPRNG 8자)를 학생이 온보딩에서 입력해 가입한다. 코드 조회는 리터럴 일치(패턴 주입 방지), 입력은 `^[A-Za-z0-9]{4,12}$` 검증.
